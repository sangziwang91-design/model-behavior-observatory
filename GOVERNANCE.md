# GOVERNANCE

> **MAINTAINER-FACING · PUBLIC-SAFE GOVERNANCE LAYER**
>
> This repository is public.
> It cannot host truly internal-only operating rules.
> This file is therefore a **sanitized maintainer governance layer**: strong enough to control release behavior, but safe enough to remain public.

## Repository role

This repository is the **public window**, not the internal source of truth.

The internal source of truth remains the reviewed Notion workflow.

## Governing principle

Nothing enters this repository unless it is already safe to circulate as a **public-safe reconstruction**.

That means every public file must be:

- non-redundant
- non-leaking
- irreversibly reconstructed
- market-readable
- evidence-linked
- safe for public circulation

## Hard release boundary

The following must **never** be published here:

- raw PDFs
- raw screenshot archives
- internal analysis pages in full
- scoring weights
- function forms
- perturbation details
- sampling derivation paths
- full protocol reconstruction paths
- routing logic
- trigger structures
- internal thresholds
- reverse-engineerable control details

## What is allowed here

The following may be published here when properly rewritten:

- findings summaries
- public-safe report snapshots
- known limits
- scope and disclosure notes
- evidence-chain summaries
- research-facing concept notes
- client-safe public excerpts that no longer expose internal reconstruction paths

## Release authority

A file is releasable only when it has already passed the upstream flow:

**Raw source → Notion review → client-safe or public-safe rewrite → GitHub**

GitHub is never the first publication surface.

## Framing control

Public language must remain:

- observation-first
- non-alarmist
- non-grandiose
- non-remediation-first

Use:

- "observation"
- "public evaluation surface"
- "confidence vs reliability"
- "known limits"
- "public-safe summary"

Avoid:

- "fixes AI"
- "solves hallucination"
- "full system"
- "complete framework disclosure"
- "internal control core"

## Change discipline

Do not update public files merely to make them sound bigger.

A public change is justified only when it improves at least one of these:

- clarity
- defensibility
- evidence linkage
- release safety
- reader navigation

## Emergency rule

If a file is later judged too revealing, the correct action is:

1. reduce exposure immediately
2. replace with safer wording
3. preserve the public-facing problem statement
4. avoid defensive over-explanation

## Maintainer reminder

This repository should accumulate **public clarity**, not public leakage.
