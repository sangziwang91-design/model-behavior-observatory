# Publish Notes

## Default stance

Public release should improve understanding.

## Publish for clarity

A public page should help a new reader do at least one of these:

- understand the problem faster
- understand the limits more clearly
- see a pattern supported by public-facing evidence
- navigate the repository more easily

If a page does none of the above, hold it for later.

## Tone rules

Public writing should remain:

- clear
- concrete
- controlled
- non-alarmist
- non-defensive

Avoid language that sounds like:

- universal proof
- final judgment
- complete disclosure
- dramatic system claims

## Stable public framing

Preferred lines:

- confidence is not the same as reliability
- this repository makes a mismatch visible
- this protocol is for observation, not remediation
- this page is a public-safe summary

## What not to publish

Do not publish pages that:

- read like raw lab notes
- expose step-by-step internals
- turn partial results into oversized claims
- require private context to be interpreted safely

## Timing rule

Do not publish only because a draft exists.

Publish when the wording is:

- stable
- readable to outsiders
- safe to circulate
- consistent with the repository role
